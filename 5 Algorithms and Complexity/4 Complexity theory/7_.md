# Quicksort Complexity - Beginner Friendly Notes

These notes explain the best, average, and worst-case performance of
Quicksort and how its best-case complexity is proven using recurrence
expansion and the Master Theorem.

------------------------------------------------------------------------

# 1. Quicksort Worst-Case Complexity (O(n²))

## Definition

The **worst case** happens when the pivot always creates one **very
large** sublist and one **empty (or nearly empty)** sublist.

Instead of dividing the problem evenly, Quicksort removes only one
element each recursive call.

### Example 1

Already sorted list (first element chosen as pivot):

    [1,2,3,4,5]

Partitions become

    []

    [2,3,4,5]

then

    []

    [3,4,5]

and so on.

The comparisons become

    4 + 3 + 2 + 1

Total

    n(n-1)/2

Time Complexity

    O(n²)

### Example 2

Reverse sorted list

    [5,4,3,2,1]

with the first element as pivot produces the same problem.

### Analogy

Imagine cutting a loaf of bread by removing **one slice at a time**. It
takes many cuts.

------------------------------------------------------------------------

# 2. Quicksort Best-Case Complexity (O(n log n))

## Definition

The **best case** occurs when the pivot is always the **median**,
splitting the array into two nearly equal halves.

### Example

    [8,3,7,1,10,5,9]

Pivot

    7

Partitions

    [3,1,5]

    [8,10,9]

Each recursive call halves the remaining work.

Time Complexity

    O(n log n)

### Analogy

Imagine repeatedly cutting a cake exactly in half.

------------------------------------------------------------------------

# 3. Quicksort Average-Case Complexity (O(n log n))

## Definition

In practice, pivots are usually neither perfect nor terrible.

Across many random inputs, partitions are reasonably balanced.

Therefore the average running time is

    O(n log n)

### Example

Some partitions:

    60% / 40%

    55% / 45%

    70% / 30%

They are not perfect but still efficient.

------------------------------------------------------------------------

# 4. Best-Case Recurrence Substitution Proof

Best-case recurrence

    T(n)=2T(n/2)+n

Expand once

    =2(2T(n/4)+n/2)+n

    =4T(n/4)+2n

Expand again

    =8T(n/8)+3n

After k expansions

    2^k T(n/2^k)+kn

Stop when

    n/2^k=1

Therefore

    k=log n

Substitute

    T(n)=n+n log n

Ignoring lower-order terms,

    O(n log n)

------------------------------------------------------------------------

# 5. Best-Case Master Theorem Proof

Recurrence

    T(n)=2T(n/2)+n

Parameters

    a=2
    b=2
    d=1

Compute

    log₂2=1

Since

    d = log_b(a)

this is the **Balanced-Work Case**.

Master Theorem gives

    O(n log n)

------------------------------------------------------------------------

# Worked Comparison

  Situation   Partition Quality     Complexity
  ----------- --------------------- ------------
  Best        Equal halves          O(n log n)
  Average     Nearly equal halves   O(n log n)
  Worst       One side empty        O(n²)

------------------------------------------------------------------------

# Real-World Analogy

Sorting books:

-   Best: Split the pile into two equal stacks each time.
-   Average: Split into roughly equal stacks.
-   Worst: Move only one book into the correct place each round.

------------------------------------------------------------------------

# Quick Summary

-   Worst Case: Bad pivots repeatedly create one large partition →
    O(n²).
-   Best Case: Median pivots halve the problem every time → O(n log n).
-   Average Case: Random pivots are usually balanced enough → O(n log
    n).
-   Substitution Proof: Expand the recurrence until depth log n.
-   Master Theorem: a=2, b=2, d=1 ⇒ Balanced case ⇒ O(n log n).
